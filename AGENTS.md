## Project Structure & Module Organization
- Project root: `ccjk-public`.
- Top-level directories detected: `assets`, `bin`, `dist`, `docs`, `evals`, `examples`, `html`, `packages`.
- Key detected files: `package.json`, `pnpm-workspace.yaml`, `README.md`, `tsconfig.json`, `.prettierrc`, `docs`.
- Runtime source lives in `src/`.
- Tests live in `tests/`.
- Release and maintenance scripts live in `scripts/`.
- Documentation lives in `docs/`.
- Avoid hand-editing generated or packaged outputs such as `dist/` unless the task explicitly targets release artifacts.

## Build, Test, and Development Commands
Use `pnpm` with Node 20+.

- `pnpm dev`: run the CLI entrypoint with `tsx`.
- `pnpm build`: build workspace release dependencies, then the root CLI bundle.
- `pnpm start`: run the local CLI via `bin/ccjk.mjs`.
- `pnpm typecheck`: build required workspace dependencies, then run TypeScript checks.
- `pnpm lint`: run ESLint.
- `pnpm test:run`: run the main Vitest suite once.
- `pnpm format`: format TypeScript source files.

## Coding Style & Naming Conventions
- Match surrounding style, indentation, naming, imports, and formatting.
- Use focused, reversible changes. Preserve unrelated user work and avoid broad reformatting.
- For TypeScript/ESM code, preserve explicit `.js` import suffixes where the repository already uses them.

## Testing Guidelines
- Run the smallest relevant verification after edits.
- Add focused regression tests when changing command behavior, permission flow, generated rules, or release surfaces.
- Run the full local gate before releases, packaging changes, or broad refactors.

## Commit & Pull Request Guidelines
- Use concise scoped commits such as `fix:`, `feat:`, `test:`, `docs:`, or `release:`.
- PRs should describe the problem, summarize the implementation, list verification commands, and include screenshots only for UI changes.

## Security & Agent-Specific Instructions
- Do not commit credentials, tokens, local auth state, dependency folders, or private runtime directories.
- Treat `clavue.md`, `AGENTS.md`, and compatibility `CLAUDE.md` as a generated rule baseline owned by `/init`; durable guidance belongs in `docs/CLAUDE-NOTES.md`.
- Finish work with changed behavior, verification run, and remaining risks.
