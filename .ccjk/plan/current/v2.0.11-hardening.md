# Feature Plan: CCJK v2.0.11 Hardening & Compatibility Upgrade

## 📋 Overview

**Objective**: Harden CCJK's Claude Code detection, installation, and configuration management to ensure robust compatibility with existing installations and prevent duplicate installations or misconfigurations.

**Expected Value**:
- Eliminate false negatives in Claude Code detection
- Prevent duplicate installations across different methods (npm, Homebrew, curl)
- Ensure configurations are written to the correct Claude Code installation
- Improve cross-platform reliability (Windows, macOS, Linux, WSL, Termux)

**Impact Scope**:
- `src/utils/installer.ts` - Major changes
- `src/utils/platform.ts` - Moderate changes
- `src/utils/config.ts` - Moderate changes
- `src/commands/init.ts` - Minor changes
- `src/utils/code-tools/codex.ts` - Minor changes

---

## 🔴 Critical Issues Identified (Must Fix)

### Issue 1: Simplistic Claude Code Detection
**Location**: `src/utils/installer.ts:15-17`
**Problem**: `isClaudeCodeInstalled()` only checks if `claude` command exists in PATH
**Risk**: HIGH - Homebrew cask installations without symlinks not detected, leading to duplicate installations

### Issue 2: Hardcoded HOME Environment Variable
**Location**: Multiple files use `process.env.HOME` directly
**Problem**: Windows uses `USERPROFILE`, not `HOME`
**Risk**: HIGH - Path detection fails on Windows when HOME is not set

### Issue 3: No File Locking for Config Operations
**Location**: `src/utils/json-config.ts`, `src/utils/ccjk-config.ts`
**Problem**: Concurrent operations could corrupt config files
**Risk**: HIGH - Data loss in concurrent scenarios

### Issue 4: Local Installation Path Never Created
**Location**: `src/utils/installer.ts:211-219`
**Problem**: Code checks for `~/.claude/local/claude` but this path is never created
**Risk**: MEDIUM - Dead code, misleading status checks

### Issue 5: Incomplete Provider Switching in Codex
**Location**: `src/utils/code-tools/codex.ts:2072-2109`
**Problem**: `switchCodexProvider()` doesn't update auth.json
**Risk**: HIGH - Authentication failures when switching providers

## 🟡 Medium Priority Issues

### Issue 6: MCP Server Overwrites Without Warning

### Issue 7: TOML Parsing Fallback Could Lose Data
**Location**: `src/utils/code-tools/codex.ts:646-681`

### Issue 8: No Backup Rotation/Cleanup
**Location**: `src/utils/config.ts:31-51`
**Problem**: Backups accumulate indefinitely

### Issue 9: Inconsistent Provider ID Sanitization
**Location**: `src/utils/code-tools/codex.ts:458-464` vs `codex-config-switch.ts:151`

### Issue 10: Windows Path Normalization Incomplete

## 🎯 Implementation Plan

### Phase 1: Critical Installation Detection Fixes (Priority 1)

#### Task 1.1: Enhanced Installation Detection
Create `getDetailedInstallationStatus()` function that:
- Detects ALL Claude Code installations (npm, Homebrew, curl, local)
- Returns installation paths, methods, and functional status
- Identifies which installation is primary (in PATH)

```typescript
interface DetailedInstallation {
  method: 'npm-global' | 'homebrew-cask' | 'homebrew-formula' | 'curl' | 'local' | 'unknown'
  path: string
  version: string | null
  inPath: boolean
  functional: boolean
}

interface DetailedInstallationStatus {
  installations: DetailedInstallation[]
  primary: DetailedInstallation | null
  configPath: string
  hasConflicts: boolean
}
```

#### Task 1.2: Fix HOME Environment Variable Usage
Replace all `process.env.HOME` with `homedir()` from `node:os`:
- `src/utils/platform.ts:213, 282, 395`
- Any other occurrences

#### Task 1.3: Remove Dead Local Installation Code
Either:
- Remove `isLocalClaudeCodeInstalled()` and related code, OR
- Implement actual local installation support

#### Task 1.4: Add Pre-Installation Duplicate Detection
Before installing, check for ALL existing installations and:
- Warn user about existing installations
- Offer to use existing installation
- Prevent duplicate installations

### Phase 2: Configuration Management Hardening (Priority 2)

#### Task 2.1: Implement Atomic File Operations
Create `atomicWriteFile()` function:
- Write to temporary file first
- Rename to target (atomic on most filesystems)
- Prevents partial writes on crash

#### Task 2.2: Add File Locking
Implement file locking for config operations:
- Use `proper-lockfile` or similar
- Wrap all config write operations
- Add timeout and retry logic

#### Task 2.3: Config Path Resolution
Create `resolveConfigPath()` function:
- Determine correct config path based on installation location
- Handle different installation methods appropriately

#### Task 2.4: Backup System Improvements
- Implement backup rotation (keep last 10)
- Verify backup integrity after creation
- Add restore functionality

### Phase 3: Codex Integration Fixes (Priority 2)

#### Task 3.1: Fix Provider Switching
Deprecate `switchCodexProvider()` or make it call `switchToProvider()`:
- Ensure auth.json is always updated
- Ensure model field is handled correctly

#### Task 3.2: Rename Ambiguous Functions
- `switchToOfficialLogin()` → `switchCodexToOfficialLogin()`
- `switchToProvider()` → `switchCodexToProvider()`
- Update all imports

#### Task 3.3: Export Sanitization Function
- Export `sanitizeProviderName()` from codex.ts
- Reuse in codex-config-switch.ts
- Remove duplicate code

### Phase 4: Cross-Platform Compatibility (Priority 3)

#### Task 4.1: Windows Path Handling
- Generalize TOML path normalization for all env variables
- Handle UNC paths
- Handle long paths (>260 chars)

#### Task 4.2: Termux Detection Enhancement
- Add PREFIX directory validation
- Verify Termux structure exists

#### Task 4.3: WSL Path Translation
- Add warning about Windows filesystem access
- Consider detecting Windows Claude Code installation

#### Task 4.4: Homebrew Path Detection
- Query `brew --prefix` for actual installation path
- Don't rely only on hardcoded paths

### Phase 5: Testing & Validation (Priority 1)

#### Task 5.1: Add Installation Detection Tests
- Multiple installation scenarios
- Broken symlink recovery
- Custom npm prefix
- WSL cross-platform

#### Task 5.2: Add Config Management Tests
- Concurrent write scenarios
- Corrupted config recovery
- Backup/restore functionality

#### Task 5.3: Add Cross-Platform Tests
- Windows path handling
- Termux environment
- WSL environment
